  <div id="readme" class="Box-body readme blob js-code-block-container p-5 p-xl-6">
    <article class="markdown-body entry-content container-lg" itemprop="text"><ul>
<li>Feature name: <code>utxo-model</code></li>
<li>Start date: 2020-05-04</li>
<li>RFC PR: <a href="https://github.com/iotaledger/protocol-rfcs/pull/0011">iotaledger/protocol-rfcs#0011</a></li>
</ul>
<h1><a id="user-content-summary" class="anchor" aria-hidden="true" href="#summary"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Summary</h1>
<p>The IOTA protocol uses an account model to keep track of the balances in the ledger state. This RFC describes and
discusses an alternative model - the UTXO model (Unspent Transaction Output) - that has several benefits over the
current model.</p>
<h1><a id="user-content-motivation" class="anchor" aria-hidden="true" href="#motivation"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Motivation</h1>
<p>Switching to a voting-based consensus requires a fast and easy way to determine a node's initial opinion for every
received transaction. This includes the ability to detect double spends and transactions that try to spend non-existing
funds.</p>
<p>The current way of calculating the ledger state - by summing up all the balance changes in a transactions past cone -
does not scale very well because it requires loading and analyzing a large amount of transactions. Furthermore, it lacks
a reliable way of detecting double spends: We would have to constantly perform tip selections and "hope" that we happen
to combine two eventually conflicting sub-tangles.</p>
<p>The UTXO model uses a different form of record keeping which enables the validation of transactions in constant time
<code>O(1)</code> and vastly improves the "expressiveness" of the ledger state by enabling things like:</p>
<ul>
<li>
<p><strong>Colored Coins:</strong> IOTA tokens can be marked with a certain "color". This color is retained throughout transfers and
gives the tokens a certain "meaning" (i.e. tokenized assets, resource- and access-tokens ...).</p>
</li>
<li>
<p><strong>Scalable Layer1 Smart Contracts:</strong> Balances can be extended by a sequence of dynamic unlock conditions that will
enable the use of a non-turing complete scripting language for smart contracts on layer1, which would not just be run
by a small committee of validators but by everybody. This will enable things like:</p>
<ul>
<li>
<p>decentralized gambling</p>
</li>
<li>
<p>decentralized exchanges</p>
</li>
<li>
<p>hashed time lock contracts (enabling i.e. "lightning network"-like scaling)</p>
</li>
<li>
<p>regulatory compliant tokenized assets</p>
</li>
<li>
<p>...</p>
</li>
</ul>
</li>
</ul>
<h1><a id="user-content-detailed-design" class="anchor" aria-hidden="true" href="#detailed-design"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Detailed design</h1>
<p>Instead of keeping track of aggregated balances per address, the <strong>Unspent Transaction (TX) Output</strong> model stores
balances associated to <strong>outputs</strong> that are the result of individual transactions moving funds and that can be addressed
and spent individually. This allows to exactly specify which funds are getting moved even if multiple parties are
funding an address in parallel.</p>
<p>A transaction moving funds consists out of three building blocks:</p>
<ul>
<li>the <code>Inputs</code> which defines a list of <code>OutputIDs</code> that reference the <em>consumed</em> <code>Outputs</code> from previous
transactions (or the genesis).</li>
<li>the <code>Outputs</code> which defines where and how many of the consumed tokens are moved.</li>
<li>the <code>Unlock Section</code> that contains data used to <em>unlock</em> the consumed inputs and <em>authorize</em> spends (usually just
the address signatures - more on that later).</li>
</ul>
<p>The following diagram shows how transactions consume outputs as inputs to create new outputs for future transactions:</p>
<p><a target="_blank" rel="noopener noreferrer" href="/hmoog/protocol-rfcs/blob/master/text/0011-utxo-model/images/utxo_flow_of_funds.png"><img src="/hmoog/protocol-rfcs/raw/master/text/0011-utxo-model/images/utxo_flow_of_funds.png" alt="sdf" style="max-width:100%;"></a></p>
<h3><a id="user-content-output-ids" class="anchor" aria-hidden="true" href="#output-ids"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Output IDs</h3>
<p>The transaction ID that creates the output together with some <em>additional identifier</em> that distinguishes different
outputs from the same transaction uniquely identifies every output.</p>
<p>Bitcoin (who introduced the UTXO model) uses the numerical index of the output inside a transaction as this
<em>additional identifier</em>. It does not actually have a concept of balances belonging to addresses, and providing a
signature for a certain address is simply a special type of <em>unlocking condition</em> for this particular output. This is a
very generic way of keeping track of balances that is to a certain degree even agnostic to the use of addresses as the
main way of interacting with such a system.</p>
<p>Even though Bitcoin has this kind of inherent agnosticism towards using addresses, it has proven to be extremely
beneficial to at least have some form of <em>logical identifier</em> that groups related outputs together and that can also be
used to <em>authorize</em> payments in the form of address signatures. It is for that very reason that the predominant part of
transaction executed on today's DLTs make use of addresses.</p>
<p>To build a bridge between the worlds of <em>UTXO-</em> and <em>account-based</em> ledgers, we define our UTXO variant to use
addresses instead of output indexes as this additional identifier:</p>
<p><code>OutputID = Pair&lt;Address, TransactionID&gt;</code></p>
<p>All <strong>colored balances</strong> sent to the same address (by the same transaction) becomes part of the same output.</p>
<h3><a id="user-content-colored-balances" class="anchor" aria-hidden="true" href="#colored-balances"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Colored Balances</h3>
<p>To be able to support tokenized assets on layer1 without the need for turing-complete smart contracts, we do not just
use a numeric value to represent a balance, but instead allow balances to have an additional <strong>color</strong> which can be
used to give coins a <em>meaning</em>.</p>
<p>A <strong>color</strong> is simply a random sequence of bytes that can be set by the user and which is retained when being
transferred:</p>
<p><code>Color = Array&lt;byte&gt;</code></p>
<p>Next to <em>user defined</em> colors, there are two <em>builtin color</em> values that carry a special meaning:</p>
<ul>
<li>
<p><code>COLOR_IOTA = Array&lt;byte&gt;(0, 0, 0, ..., 0)</code> represents the <em>base color</em> of uncolored coins.</p>
</li>
<li>
<p><code>COLOR_MINT = Array&lt;byte&gt;(255, 255, 255, ..., 255)</code> represents a color which is being replaced by the
<code>Transaction ID</code> before being booked as an output. It is consequently used to "mint" new colored coins.</p>
</li>
</ul>
<p>A <strong>colored balance</strong> is accordingly a combination of a numeric balance with its corresponding color:</p>
<p><code>ColoredBalance = {Balance: uint64, Color: Color}</code></p>
<h3><a id="user-content-outputs" class="anchor" aria-hidden="true" href="#outputs"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Outputs</h3>
<p>An Output is a container for the colored balances that were spent by a particular transaction to a certain address. In
addition to the list of colored balances and its ID, we store an <code>OPCode</code> and a corresponding <code>OPCodeMetadata</code>.</p>
<p><code>Output = {ID: OutputID, Balances: Array&lt;ColoredBalance&gt;, OpCode: OPCode, OPCodeMetadata: OPCodeMetadata}</code></p>
<h3><a id="user-content-opcodes" class="anchor" aria-hidden="true" href="#opcodes"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>OPCodes</h3>
<p>Currently, an output is <em>unlocked</em> if there is a valid signature for the address that holds the output, and it is not
possible to define alternative conditions.</p>
<p>There are however use cases where being able to define different conditions might be useful. A famous example are hash
locks and time locks, that are used to create
<a href="https://en.bitcoinwiki.org/wiki/Hashed_Timelock_Contracts" rel="nofollow">Hashed Timelock Contracts</a> enabling things like atomic swaps
and 2nd layer scaling solutions like the Lightning Network.</p>
<p>The opcode is a single byte that is mapped to a specific unlock method. This will allow us to switch between different
unlock conditions which introduces some very basic smart contract capabilities on the base layer.</p>
<p><code>OPCode = byte</code></p>
<p>Since some opcodes might require certain parameters (like a timestamp or a hash), we define a metadata field, that holds
a marshaled version of these parameters.</p>
<p><code>OPCodeMetadata = Array&lt;byte&gt;</code></p>
<p>As long as these checks are not much more <em>expensive</em> than signature checks, it does not introduce problems but
massively extends the functionality of the protocol. We have to however be very careful with adding new conditions, to
not accidentally introduce potential bottlenecks.</p>
<p>In contrast to bitcoin, we do not add a fully scriptable outputs (because that could result in long execution times),
but we offer predefined methods for different use cases.</p>
<p>For now, we simply reserve the additional fields in the output but hard code the OPCode to be <code>SIG_CHECK = OPCode(0)</code>.</p>
<h3><a id="user-content-conflict-detection" class="anchor" aria-hidden="true" href="#conflict-detection"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Conflict Detection</h3>
<p>The biggest benefits of this ledger state is the fast and easy detection of double spends and invalid transaction:</p>
<ul>
<li>
<p>A transaction is valid, if the sum of the moved funds is <code>0</code> and it only spends colors in the amounts that were
available in the used inputs. It is however possible to override the color of the consumed tokens using either the
<code>MINT_COLOR</code> color to create a new colored token or by using the <code>IOTA_COLOR</code> to remove the color.</p>
<p>Instead of having to walk through multiple transaction to eventually find the spent balance in the tangle, we can now
instantly load the corresponding funds by accessing the consumed outputs. A transaction that tries to spend non-existing
funds would never become solid.</p>
</li>
<li>
<p>Accordingly, a transaction is performing a double spend when there is already another transaction consuming the same
output.</p>
</li>
</ul>
<h1><a id="user-content-drawbacks" class="anchor" aria-hidden="true" href="#drawbacks"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Drawbacks</h1>
<ul>
<li>
<p>A UTXO-based ledger is much more complex in the implementation than just a dictionary of balances.</p>
</li>
<li>
<p>A UTXO-based ledger stores more information and therefore requires "potentially" more disk space.</p>
<p>However, if
sweeping becomes a central part of the protocol with all wallets supporting it, then this solution will most
probably not perform much worse than an account-based ledger.</p>
<p>Considering how we store ledger state today, with several <em>checkpoints</em> where we <em>cache the ledger state</em> to reduce
possible tangle walking time, it might also be possible that the resource consumption drops because we are no longer
required to have these additional checkpoints with their corresponding ledger state (in memory and on disk).</p>
<p>In general, it does not introduce a new attack vector because it is already today possible to spread out small
balances onto many addresses, which is the equivalent of spreading a lot of outputs on a single address.</p>
</li>
</ul>
<h1><a id="user-content-rationale-and-alternatives" class="anchor" aria-hidden="true" href="#rationale-and-alternatives"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Rationale and alternatives</h1>
<p>There are currently only two options for a ledger state, that are known - the UTXO model, and the account model:</p>
<ul>
<li>
<p>The account-model does not work very well with "parallelism" (i.e. multiple people sending funds to the same
address). We can never be sure, that our transaction has seen all the funds and the only way to make sure, that this is
the case is to use the tangle structure to define a <em>scope</em> for a spend. This however is very slow, because the only way
to use the tangle structure is to walk around in the tangle. In addition, it adds additional constraints to the tip
selection as it requires to have the correct funding transactions in the past cone of a spending transaction.</p>
</li>
<li>
<p>Considering the "parallel" nature of the tangle, it seems like UTXO is the only reasonable choice. Using addresses as
the additional identifier for outputs, allows us to bundle multiple colored balances in a single output, which leads to
a smaller fragmentation of the funds as if we would give every color a separate output.</p>
</li>
</ul>
<p>If we decide to not go with this approach, then we will essentially limit ourselves to just a few hundred TPS because of
the massive overhead of checking incoming transactions. In addition, the conflict detection will be much harder.</p>
<h1><a id="user-content-unresolved-questions" class="anchor" aria-hidden="true" href="#unresolved-questions"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Unresolved questions</h1>
<ul>
<li>We need to think about good unlock opcodes and options.</li>
<li>We should define a PR that specifies the "branch based ledger state" as this is very closely related to the
use of UTXO.</li>
</ul>
